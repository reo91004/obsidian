# 0️⃣ Introduction to [[Physical Attacks]]

**Physical Attacks and Countermeasures**는 암호 장치의 물리적 특성을 이용하여 비밀 키를 추출하는 공격 기법과 그에 대한 방어책을 다루는 분야이다. 이 세션에서는 암호학의 기초, 암호 장치의 정의, 물리적 공격의 기본 개념, 그리고 하드웨어 설계의 기초를 다룬다.

## 왜 Physical Attacks인가?

1. **암호 알고리즘 자체는 안전**: AES, RSA 등 현대 암호 알고리즘은 수학적으로 안전하다고 증명되어 있다.
2. **구현(Implementation)이 문제**: 알고리즘이 아무리 안전해도, 그것을 **구현한 장치**는 물리적 특성(전력 소비, 실행 시간, 전자기 방사 등)을 통해 비밀 정보를 누출할 수 있다.
3. **실생활 위협**: 뱅킹 카드, 자동차 키 포브, 여권, ID 카드 등 우리가 일상적으로 사용하는 거의 모든 암호 장치가 공격 대상이 될 수 있다.
4. **인증 및 규제**: BSI(독일 연방정보보안청) 등에서 암호 장치에 대한 물리적 공격 저항성을 인증 요구사항으로 규정하고 있다.

# 1️⃣ [[Cryptography]] 기초

## 1. 암호학의 정의

암호학(Cryptography)은 단순히 "비밀 글쓰기의 과학"이라고 정의하기엔 너무 단순하다. 보다 정확한 정의는:

> **디지털 정보, 트랜잭션, 그리고 연산을 보호하기 위한 기술의 과학적 연구**

## 2. 암호학의 4가지 요구사항

| 요구사항 | 설명 |
|----------|------|
| **[[Integrity]]** (무결성) | 수신자가 통신 중 데이터가 변경되었는지 감지할 수 있어야 한다. |
| **[[Confidentiality]]** (기밀성) | 통신을 도청하는 제3자가 전송 내용을 이해할 수 없어야 한다. |
| **[[Authenticity]]** (인증성) | 수신자가 송신자가 누구인지 확인할 수 있어야 한다. 또는 송신자가 "내가 이것을 보냈다"를 증명할 수 있어야 한다. |
| **[[Non-repudiation]]** (부인 방지) | 서명한 사람이 "나는 그것을 서명하지 않았다"고 부인할 수 없어야 한다. (인증의 반대 방향) |

각각의 암호학적 프리미티브(Cryptographic Primitive)는 이 요구사항 중 하나 또는 여러 개를 충족시킨다. 필요에 따라 다른 종류의 프리미티브를 사용한다.

## 3. 고전적 암호 시나리오

```
Alice ←──── [Insecure Channel] ────→ Bob
                    ↑
                    │ (도청자)
                    Eve
```

- **Alice**와 **Bob**이 통신하려 한다.
- 통신 채널은 **보안되지 않음** (Eve가 도청 가능).
- **대칭 암호**(예: AES)를 사용하면 양쪽에 **동일한 키**가 있어야 한다.
- Alice가 키로 암호화하고, Bob이 **같은 키**로 복호화한다.
- 이 강의에서는 주로 **Confidentiality**(기밀성)와 **키의 비밀 유지**를 다룬다.

## 4. [[Kerckhoffs' Principle]] (케르크호프스의 원리)

> **"암호 시스템은 비밀일 필요가 없으며, 적의 손에 들어가더라도 불편함이 없어야 한다."**

### 의미

- 암호 알고리즘의 **세부 사항은 공개**되어야 한다.
- **오직 키만이 비밀**이어야 한다.
- "나는 알고리즘을 공개하지 않겠다, 그래서 안전하다"는 **허용되지 않는다**.

### 왜 이 원리가 중요한가?

1. **키 관리가 쉬움**: 알고리즘 전체를 비밀로 유지하는 것보다 짧은 키 하나를 비밀로 유지하는 것이 훨씬 쉽다.
2. **키 변경 용이**: 키가 유출되었다고 의심되면 언제든지 키만 교체하면 된다. 알고리즘 전체를 교체할 필요 없음.
3. **세션 키 사용 가능**: 키를 주기적으로 변경하여 보안을 강화할 수 있다.
4. **다자간 통신 용이**: 같은 알고리즘을 사용하면서 각 통신 쌍마다 다른 키를 사용할 수 있다.
5. **공개 검토(Public Review)**: 알고리즘을 공개하면 전 세계 연구자들이 검토하고 취약점을 발견할 수 있다.

### AES의 예

- **AES**는 NIST(미국 국립표준기술연구소)가 표준화했지만, NIST가 설계한 것이 아니다.
- 공개 **경쟁(Competition)**을 통해 여러 알고리즘이 제출되었다.
- 전 세계 연구자들이 **공개 검토**를 수행했다.
- 최종적으로 **Rijndael** 알고리즘이 AES로 선정되었다.
- **지금도** AES에 대한 연구와 보안 평가가 계속되고 있다.

### Security by Obscurity의 문제

알고리즘을 비밀로 유지하려는 회사들의 사례:

1. 알고리즘을 비밀로 유지하거나 **NDA(비밀유지계약)**로 보호한다.
2. 공개 검토가 불가능하다.
3. 어느 날 알고리즘이 **유출**된다.
4. 해당 분야 전문 연구자들이 **취약점을 발견하고 공개**한다.

> **결론**: Security by Obscurity는 과학 커뮤니티에서 **허용되지 않는다**.

## 5. 보안의 정의

> **"주어진 작업에 대한 암호 체계가 안전하다"**는 것은 **"특정 능력을 가진 공격자가 특정 목표를 달성할 수 없다"**는 의미이다.

### Adversary Model의 중요성

- "이 알고리즘은 안전하다"라고만 말할 수 없다.
- **반드시 공격자 모델(Adversary Model)을 정의**해야 한다.
- 예: "AES-128은 안전하다" → **어떤 공격자 모델에서?**

### AES-128의 예

- **모든 가능한 키를 탐색**하면 $2^{128}$개의 경우의 수.
- 공격자가 $2^{128}$개를 모두 탐색할 수 있다면 → **안전하지 않음**.
- 현재 기술 수준에서 공격자는 대략 **$2^{90}$** 정도의 탐색 공간만 처리 가능하다고 가정.
- 이 공격자 모델 하에서 → **AES는 안전하다**.

> **Physical Attacks에서도 마찬가지**: 공격자 모델을 정의해야만 보안을 증명하거나 주장할 수 있다.

# 2️⃣ [[Cryptographic Device]]란?

## 1. 정의

**Cryptographic Device(암호 장치)**는:

- 암호 알고리즘을 **구현**하고
- 암호 키를 **저장**하는 전자 장치이다.

## 2. 예시

| 장치 | 설명 |
|------|------|
| **스마트폰** | 내부에 AES, RSA, ECC 등을 처리하는 암호 코프로세서가 있다. |
| **차고 문 리모컨** | 버튼을 누르면 문이 열리고 닫힌다. 이 통신은 암호화되어야 한다. |
| **자동차 키 포브** | 버튼을 누르면 중앙 잠금장치가 작동한다. |
| **뱅킹 카드 (EC 카드)** | 칩이 내장되어 있으며, 암호 프리미티브를 포함한다. |
| **여권, ID 카드** | 암호 프리미티브를 포함하며 물리적 공격에 대한 보안이 필요하다. |
| **[[Immobilizer]]** | 자동차 키 내부의 칩으로, ECU와 무선 통신한다. 기계적으로 키를 복제해도 이 칩 없이는 시동이 걸리지 않는다. |
| **USB 토큰** | 일부는 암호 프리미티브를 포함한 보안 USB이다. |
| **[[RFID]] 태그** | 교통카드 등. 무선 통신이므로 보안이 필요하다. 안테나를 통해 전력을 받아 동작하며, 내부에 스마트카드 칩이 있다. |

## 3. 암호 장치의 보안 원칙

암호 알고리즘과 마찬가지로:

> **암호 장치의 보안도 구현의 비밀에 의존해서는 안 된다.**

- "나는 AES를 구현했지만 어떻게 구현했는지는 비밀이다. 그래서 키를 복구할 수 없다" → **허용되지 않음**.
- 이것도 **Security by Obscurity**이다.

### 그럼에도 일부 회사가 Obscurity를 사용하는 이유

- **Reverse Engineering을 어렵게 만들기 위해**.
- 공격자가 해야 할 작업에 **한 단계를 추가**하는 효과.
- 공격자는 먼저 **Reverse Engineering**을 수행하여 구현 방식을 파악한 후, 물리적 공격을 시도해야 한다.

### 중요한 구분

- **암호 장치를 깨는 것** ≠ **암호 알고리즘을 깨는 것**
- 예: 자동차 키 포브의 AES 구현을 물리적 공격으로 깼다.
- 이것은 **해당 구현에서 사용된 키를 복구**한 것이지, **AES 알고리즘 자체를 깬 것이 아니다**.
- 키가 주기적으로 변경되면, 복구한 키는 **해당 기간에만 유효**하다.

> **암호 프리미티브의 보안은 오직 키의 비밀에 기반한다. 키가 복구되면 보안은 완전히 무너진다.**

## 4. Pay-TV 시스템: Physical Attacks의 역사적 기원

Physical Attacks가 과학 커뮤니티에 등장하게 된 계기는 **Pay-TV 시스템**이다.

### 배경

- 20년 이상 전, 위성 수신기에 **스마트카드**를 넣어 유료 채널을 시청했다.
- 스마트카드는 뱅킹 카드와 유사한 구조: **마이크로컨트롤러 + 핀**으로 수신기와 통신.
- 해커 커뮤니티에서 이 카드를 **해킹하여 복제**하려는 시도가 활발했다.

### 해킹 과정

1. 단순히 바이너리를 복사하는 것으로는 작동하지 않음 (암호화됨).
2. 암호 알고리즘이 내부에 있고, **세션 키**도 변경됨.
3. **물리적 공격**을 통해 키를 추출.
4. 빈 카드에 프로그래밍하여 불법 복제.
5. 인터넷에 바이너리 파일이 공유되어 누구나 불법 시청 가능.

### 현재

- 이 비즈니스 모델은 더 이상 유효하지 않음 (인터넷 기반 스트리밍으로 전환).
- 하지만 Physical Attacks는 **다른 많은 장치에 여전히 적용 가능**.

## 5. BSI 인증

**BSI (Bundesamt für Sicherheit in der Informationstechnik)**: 독일 연방정보보안청

### 인증 프로세스

1. BSI가 암호 장치에 대한 **보안 요구사항(Criteria)**을 정의한다.
2. BSI가 인증한 **평가 연구소(Evaluation Labs)**가 장치를 평가한다.
3. 예: 뱅킹 카드 제조 회사가 카드를 만든다.
4. 은행이 이 카드를 구매하여 고객에게 배포하려면, 카드가 **평가 연구소의 평가**를 통과해야 한다.
5. 평가 결과는 **BSI의 인증**을 받아야 한다.
6. 독일에서 판매되는 모든 뱅킹 카드는 **최소한 설계 단계에서 BSI 인증**을 받았다.

> 이러한 인증 제도는 **20년 전에는 존재하지 않았다**. 연구자들이 위협을 발견하고, 산업계가 대응책을 마련하면서 발전해왔다.

# 3️⃣ Physical Attacks 예시: USB Stick PIN 검증

## 1. 시나리오 설정

- 누군가 **비밀 데이터가 담긴 USB 스틱**을 발견했다.
- USB 스틱은 **8자리 PIN**으로 보호되어 있다.
- PIN을 입력해야 내용에 접근할 수 있다.

### Brute Force 접근법

- 8자리 PIN의 가능한 조합: $10^8 = 100,000,000$개
- 각 PIN 검증에 1초가 걸린다고 가정.
- 장치가 10회 실패 후 잠기지 **않는다고** 가정. (Open-minded해야 함 - 이런 장치도 존재)
- **소요 시간**: 약 **3년**

$$
\frac{10^8 \text{ seconds}}{60 \times 60 \times 24 \times 365} \approx 3.17 \text{ years}
$$

> 데이터가 3년 후에는 이미 **가치가 없을 수** 있다.

## 2. [[Timing Attack]]: PIN 검증의 취약점

### 마이크로프로세서의 PIN 검증 로직

소프트웨어 엔지니어가 PIN 검증을 구현하는 일반적인 방식:

```c
// pin_check.c
bool check_pin(char* input, char* stored) {
    for (int i = 0; i < 8; i++) {
        if (input[i] != stored[i]) {
            return false;  // 틀린 자리를 발견하면 즉시 종료
        }
    }
    return true;
}
```

**`strcomp`** 또는 **`memcmp`** 함수도 유사하게 동작:

```
for i = 1 to 8:
    if S1[i] != S2[i]:
        return false
return true
```

### 취약점: Data-Dependent Execution Time

- **첫 번째 자리가 틀리면**: 즉시 반환 → **짧은 시간**
- **첫 번째 자리가 맞고 두 번째가 틀리면**: 루프를 한 번 더 돌고 반환 → **조금 더 긴 시간**
- **모든 자리가 맞으면**: 루프를 8번 돌고 반환 → **가장 긴 시간**

> **실행 시간이 올바른 자릿수에 의존**한다!

### 공격 가정

| 가정 | 설명 |
|------|------|
| **시간 측정 가능** | 공격자는 PIN 입력부터 응답까지의 시간을 측정할 수 있다. |
| **높은 정밀도 필요** | 마이크로프로세서의 클럭이 빠르면 고정밀 카운터가 필요하다. |
| **노이즈 존재** | 측정에 노이즈가 있을 수 있으므로 **여러 번 측정하고 평균**을 낸다. |

### [[Divide and Conquer]] 공격 절차

**핵심 아이디어**: 8자리 전체를 한 번에 찾지 말고, **한 자리씩 개별적으로** 찾는다.

#### Step 1: 첫 번째 자리 찾기

```
00000000 → 시간 T0 측정 (여러 번 평균)
10000000 → 시간 T1 측정
20000000 → 시간 T2 측정
...
90000000 → 시간 T9 측정
```

- **가장 긴 시간**을 가진 첫 번째 자리가 **정답**.
- 예: T3이 가장 길면 → 첫 번째 자리는 **3**.

#### Step 2: 두 번째 자리 찾기

첫 번째 자리를 3으로 **고정**하고 반복:

```
30000000 → 시간 측정
31000000 → 시간 측정
32000000 → 시간 측정
...
39000000 → 시간 측정
```

#### 계속 반복

- 이 방식으로 8자리 모두를 찾는다.
- **노이즈가 없다면**: $10 \times 8 = 80$번의 측정으로 충분.
- 노이즈가 있다면: 각 경우를 **여러 번 측정하고 평균**.

### 복잡도 비교

| 방식 | 시도 횟수 |
|------|-----------|
| Brute Force | $10^8 = 100,000,000$ |
| Timing Attack (Divide and Conquer) | $10 \times 8 = 80$ |

> **물리적 공격이 작동하는 이유**: 키 전체를 한 번에 찾지 않고, **개별 부분을 하나씩** 찾기 때문이다.

## 3. 대응책 시도와 한계

### 시도 1: 항상 모든 자리 검사

```c
// pin_check_v2.c
bool check_pin(char* input, char* stored) {
    bool result = true;
    for (int i = 0; i < 8; i++) {
        if (input[i] != stored[i]) {
            result = false;  // 즉시 반환하지 않음
        }
    }
    return result;
}
```

**문제점**: 여전히 취약할 수 있다!

- `result = false` 명령이 실행되는 경우와
- 아무것도 실행되지 않는 경우의 **클럭 사이클이 다르다**.

### 시도 2: else 절에 NOP 추가

```c
// pin_check_v3.c
bool check_pin(char* input, char* stored) {
    bool result = true;
    for (int i = 0; i < 8; i++) {
        if (input[i] != stored[i]) {
            result = false;
        } else {
            asm("nop");  // No Operation
        }
    }
    return result;
}
```

**문제점**:

1. **컴파일러 최적화**가 `nop`를 제거할 수 있다.
2. 어셈블리 수준에서 **다른 명령어 시퀀스**가 실행된다.

> **결론**: 고수준 코드만 보면 안 된다. **어셈블리 코드**와 **각 명령어의 클럭 사이클**을 확인해야 한다.

### 핵심 교훈

> **Constant-Time(상수 시간) 구현**은 매우 어렵다. 어셈블리 수준에서 검증해야 하며, 하드웨어 특성까지 고려해야 한다.

## 4. [[Power Analysis Attack]]: 시간이 일정해도 공격 가능

### 시나리오

- Timing Attack 대응책을 **완벽하게** 구현했다고 가정.
- 실행 시간이 **완전히 일정**하다.
- **그래도 공격이 가능한가?**

### CMOS 기술과 전력 소비

- 대부분의 디지털 회로는 **CMOS 기술**로 구현된다.
- CMOS 회로는 **동작할 때 에너지를 소비**한다.
- **전력 소비량**은 **처리하는 데이터**와 **실행하는 명령어**에 의존한다.

### 공격 원리

```
       ↑ Power
       │    ___
       │   /   \___    ← result = false 실행 시
       │  /
       │ /
       └──────────────→ Time

       ↑ Power
       │       ___
       │   ___/   \    ← nop 실행 시
       │  /
       │ /
       └──────────────→ Time
```

- `result = false`를 실행할 때와 `nop`를 실행할 때의 **전력 소비 패턴이 다르다**.
- 아주 작은 차이지만, **비교하면 구분할 수 있다**.

### 공격 절차

1. 각 PIN 후보에 대해 **전력 소비 곡선(Power Trace)**을 측정한다.
2. 노이즈가 있으면 **여러 번 측정하고 평균**을 낸다.
3. 전력 소비 패턴을 비교한다.
4. **다른 패턴**을 보이는 것이 정답 자리를 나타낸다.

> Timing Attack과 **완전히 동일한 Divide and Conquer 방식**으로 공격한다.

### 대응의 어려움

- 전력 소비를 **데이터와 독립적**으로 만드는 것은 매우 어렵다.
- 이것이 **본 강의의 주요 주제** 중 하나이다.

## 5. [[Fault Injection Attack]]: 적극적 개입

### 시나리오

- Timing Attack과 Power Analysis Attack을 **모두 방어**했다고 가정.
- **그래도 공격이 가능한가?**

### 디지털 회로의 정상 동작 조건

디지털 회로가 **올바르게 동작**하려면 특정 조건이 필요하다:

| 조건 | 설명 |
|------|------|
| **Clock Frequency** | 클럭 주기 $T$가 **Critical Path Delay** $T_{cp}$보다 커야 한다. |
| **Temperature** | 정해진 온도 범위 내에서 동작해야 한다. (예: 최대 80°C) |
| **Voltage ($V_{DD}$)** | 정해진 전압을 공급해야 한다. (기술에 따라 0.8V ~ 1.2V 등) |

### Critical Path Delay 설명

```
┌─────────┐     ┌──────────────────┐     ┌─────────┐
│ Register│────→│ Combinational    │────→│ Register│
│   (FF)  │←────│     Circuit      │←────│   (FF)  │
└─────────┘     └──────────────────┘     └─────────┘
     ↑                                        ↑
     └────────────── Clock ───────────────────┘
```

- **Mealy/Moore Machine** 구조.
- 조합 회로(Combinational Circuit)의 **가장 긴 경로**의 지연 시간 = **Critical Path Delay**.
- 클럭이 이보다 빠르면 → 조합 회로가 완료되기 전에 레지스터가 값을 저장 → **잘못된 데이터 저장**.

### Fault Injection 방법

| 방법 | 설명 |
|------|------|
| **Clock Glitch** | 클럭을 순간적으로 빠르게 만든다. |
| **Voltage Glitch (Power Glitch)** | 전압을 순간적으로 낮추거나 높인다. |
| **Temperature** | 칩을 가열하여 타이밍을 변화시킨다. |

### 기술 노드와 전압의 관계

| 기술 노드 | $V_{DD}$ | 특징 |
|-----------|----------|------|
| 28nm | 1.0 ~ 1.2V | |
| 16nm | ~1.0V | |
| 6nm | ~0.8V | 낮은 전압 → 커패시터 충/방전 시간 단축 → 더 빠른 동작 가능 |

- 기술이 발전할수록 **트랜지스터가 작아지고**, **$V_{DD}$가 낮아진다**.
- $V_{DD}$가 낮으면 → 커패시터 충/방전이 빨라짐 → **더 빠른 클럭** 가능 + **전력 소비 감소**.

### 마이크로컨트롤러에서의 Fault Injection 효과

- Clock Glitch나 Voltage Glitch를 주입하면 → **명령어가 스킵**될 수 있다.
- 이것은 마이크로프로세서의 **일반적인 동작**이다 (나중에 자세히 학습).

### USB Stick 공격 시나리오

1. PIN 검증 함수 `check_pin()`이 호출되는 시점에 **Fault를 주입**한다.
2. 함수 호출 명령어가 **스킵**된다.
3. PIN 검증이 **완전히 무시**된다.
4. **비밀 데이터에 접근 성공**.

### Fault Injection의 특징

- **매번 접근할 때마다** Fault를 주입해야 한다.
- 또는 **한 번 접근하여 데이터를 복사**하면 된다.
- **정확한 타이밍**을 알아야 하거나, **여러 시점을 시도**해야 한다.

> **공격자는 PIN을 알아내지 못했지만, 인증 과정을 우회하여 데이터에 접근**했다.

# 4️⃣ Physical Attacks의 분류

## 1. 공격자 행동에 따른 분류

| 분류 | 설명 |
|------|------|
| **[[Passive Attack]]** | 장치를 **관찰만** 한다. 장치는 공격당하고 있음을 알기 어렵다. (예: 시간 측정, 전력 측정) |
| **[[Active Attack]]** | 장치를 **방해/조작**한다. (예: Fault Injection) |

## 2. 침습성에 따른 분류

| 분류 | 설명 |
|------|------|
| **[[Invasive Attack]]** | 공격자에게 **제한이 없다**. 칩을 열고(Decap), 전기적 접촉을 만들고, 구멍을 뚫을 수 있다. |
| **[[Semi-Invasive Attack]]** | 칩을 **열 수 있지만**, 전기적 접촉은 **만들 수 없다**. 광학 관찰, 레이저 조사 등 가능. |
| **[[Non-Invasive Attack]]** | 칩을 **열 수 없다**. 외부에서만 관찰하거나 조작한다. |

### Non-Invasive가 중요한 이유

- 일부 장치는 **Tamper Protection**이 있어서 열면 **자동으로 비활성화**된다.
- Non-Invasive 공격은 장치가 **공격당하고 있음을 감지하기 어렵다**.

## 3. 6가지 공격 유형

| | Passive | Active |
|---|---------|--------|
| **Invasive** | **Probing**: 칩을 열고 내부 신호선에 마이크로프로브를 연결하여 값을 읽는다. | **Forcing Permanent Faults**: 칩을 열고 전기적 접촉으로 영구적 결함을 만든다. |
| **Semi-Invasive** | **Optical Inspection / Photon Emission**: 칩을 열고 동작 중 방출되는 광자를 특수 카메라로 관찰하여 어떤 회로가 토글되는지 파악한다. | **Laser Fault Injection**: 칩을 열고 레이저 빔을 매우 작은 영역에 조사하여 Fault를 주입한다. |
| **Non-Invasive** | **[[Side-Channel Analysis]] (SCA)**: 칩을 열지 않고 전력 소비, 실행 시간, 전자기 방사 등을 측정한다. 장치가 공격을 감지하기 매우 어렵다. | **Fault Injection (Clock/Power Glitch, Temperature)**: 칩을 열지 않고 외부에서 클럭, 전압, 온도를 조작하여 Fault를 주입한다. |

## 4. 본 강의의 범위

```
                    Passive              Active
              ┌─────────────────┬─────────────────┐
  Invasive    │    Probing      │  Perm. Faults   │
              ├─────────────────┼─────────────────┤
Semi-Invasive │ Photon Emission │ Laser Fault Inj │
              ├─────────────────┼─────────────────┤
 Non-Invasive │ ★ SCA (주로)   │ ★ Fault Inj    │
              └─────────────────┴─────────────────┘
```

- 본 강의는 주로 **Non-Invasive Passive (Side-Channel Analysis)**를 다룬다.
- **Non-Invasive Active (Fault Injection)**도 일부 다룬다.
- Invasive와 Semi-Invasive는 다루지 않는다.

### Side-Channel Analysis가 가장 위험한 이유

> **장치가 공격당하고 있음을 감지하기 매우 어렵다.** 전력 소비나 실행 시간을 측정하는 것을 장치가 알 방법이 거의 없다.

# 5️⃣ Hardware Design 기초

## 1. 칩 설계 흐름

```
┌──────────────────┐
│   Specification  │ ← 장치가 무엇을 해야 하는지 정의
└────────┬─────────┘
         ↓
┌──────────────────┐
│ High-Level Design│ ← Behavioral Design (RTL)
│   (VHDL/Verilog) │   HDL 코드 작성
└────────┬─────────┘
         ↓
┌──────────────────┐
│    Synthesis     │ ← Standard Cell Library 사용
│                  │   HDL → Gate-level Netlist
└────────┬─────────┘
         ↓
┌──────────────────┐
│  Floor Planning  │ ← 컴포넌트 배치 계획
└────────┬─────────┘
         ↓
┌──────────────────┐
│    Placement     │ ← 실제 위치 결정
└────────┬─────────┘
         ↓
┌──────────────────┐
│     Routing      │ ← 연결선 배치
└────────┬─────────┘
         ↓
┌──────────────────┐
│   Fabrication    │ ← Fab에서 칩 제조
│    (Tape-out)    │
└──────────────────┘
```

- **각 단계에서 Verification(검증)**을 수행한다.
- 요구사항을 충족하지 못하면 **이전 단계로 돌아가서 수정**한다.

## 2. [[Standard Cell Library]]

### Synthesis란?

- **Behavioral HDL 코드** (예: VHDL, Verilog)를 입력받는다.
- **Standard Cell Library**를 참조하여 **Gate-level Netlist**를 생성한다.

### Standard Cell Library 내용

```
┌─────────────────────────────────────────┐
│         Standard Cell Library           │
├─────────────────────────────────────────┤
│ AND2: 2-input AND gate                  │
│   - Area: 2 units                       │
│   - Delay: 0.1ns                        │
│   - Power: 0.5uW                        │
├─────────────────────────────────────────┤
│ AND3: 3-input AND gate                  │
│   - Area: 3 units                       │
│   - Delay: 0.12ns                       │
│   - Power: 0.6uW                        │
├─────────────────────────────────────────┤
│ OR2, OR3, XOR2, NAND2, NOR2, ...        │
│ FF (Flip-Flop), MUX, ...                │
└─────────────────────────────────────────┘
```

- 각 셀의 **면적, 지연 시간, 전력 소비** 정보가 포함되어 있다.
- Synthesizer가 사용자의 요구사항(예: "3GHz 클럭 지원")에 맞게 셀을 선택한다.

### ASIC vs FPGA

| | ASIC | FPGA |
|---|------|------|
| **Library** | Standard Cell Library | FPGA 내장 컴포넌트 |
| **구성 요소** | 작은 게이트 (2-input AND 등) | 큰 게이트 (6-input LUT 등) |
| **구조** | 완전 맞춤 설계 | 고정된 구조를 설정(configure) |
| **유연성** | 제조 후 변경 불가 | 재프로그래밍 가능 |

## 3. Layout과 Netlist

- **Netlist**: 컴포넌트들이 **어떻게 연결**되는지를 나타내는 그래프.
- **Layout**: **물리적 위치**와 **연결선의 경로**를 나타내는 기하학적 기술.

## 4. 스마트카드 칩의 구조

```
      ┌─────────────────────────────────┐
      │  ┌─┐  ┌─┐  ┌─┐  ┌─┐  ┌─┐  ┌─┐  │ ← Contact Pads (7~8개)
      │  └─┘  └─┘  └─┘  └─┘  └─┘  └─┘  │   (카드 앞면에서 보이는 금색 부분)
      │         ╲   │   ╱              │
      │          ╲  │  ╱               │ ← Bonding Wires
      │           ╲ │ ╱                │   (패드와 칩을 연결)
      │    ┌───────────────┐           │
      │    │               │           │
      │    │  Silicon Die  │           │ ← 실제 회로가 있는 실리콘 칩
      │    │               │           │   (매우 작음)
      │    └───────────────┘           │
      └─────────────────────────────────┘
               ↑
           카드 뒷면
```

- **Contact Pads**: 특정 프로토콜로 외부 장치와 통신.
- **Contactless 통신**: 같은 칩에 **안테나**를 연결하여 무선으로 전력을 받고 통신.

# 6️⃣ [[AES]] 구조 및 구현

## 1. AES 개요

| 항목 | AES-128 | AES-192 | AES-256 |
|------|---------|---------|---------|
| **Key Size** | 128 bits | 192 bits | 256 bits |
| **Block Size** | 128 bits | 128 bits | 128 bits |
| **Rounds** | 10 | 12 | 14 |

- 본 강의에서는 주로 **AES-128**을 다룬다.
- **Plaintext**와 **Ciphertext**는 항상 **128 bits**.

## 2. AES-128 구조

```
Plaintext (128 bits)
       │
       ↓
┌──────────────────┐
│   AddRoundKey    │ ← Round Key 0 (= 원래 키)
└────────┬─────────┘
         │
    ┌────┴────┐
    │         │ × 9 rounds
    ↓         │
┌──────────┐  │
│ SubBytes │  │
└────┬─────┘  │
     ↓        │
┌──────────┐  │
│ ShiftRows│  │
└────┬─────┘  │
     ↓        │
┌──────────┐  │
│MixColumns│  │
└────┬─────┘  │
     ↓        │
┌──────────┐  │
│AddRoundKey│  │
└────┬─────┘  │
     │        │
     └────────┘
         │
    10th round (마지막 라운드)
         │
         ↓
┌──────────────────┐
│    SubBytes      │
└────────┬─────────┘
         ↓
┌──────────────────┐
│   ShiftRows      │
└────────┬─────────┘
         ↓
┌──────────────────┐  ← MixColumns 없음!
│   AddRoundKey    │
└────────┬─────────┘
         ↓
   Ciphertext (128 bits)
```

### Round Keys

- **Key Schedule** 함수가 128-bit 원래 키로부터 **11개의 Round Keys**를 생성한다.
- Round Key 0 = 원래 키
- Round Key 1 ~ 10 = Key Schedule로 생성

## 3. State Matrix (4×4 바이트 행렬)

AES의 모든 연산은 **4×4 바이트 행렬** 형태로 처리된다:

$$
\text{State} = \begin{pmatrix}
s_{0,0} & s_{0,1} & s_{0,2} & s_{0,3} \\
s_{1,0} & s_{1,1} & s_{1,2} & s_{1,3} \\
s_{2,0} & s_{2,1} & s_{2,2} & s_{2,3} \\
s_{3,0} & s_{3,1} & s_{3,2} & s_{3,3}
\end{pmatrix}
$$

- 각 $s_{i,j}$는 **8 bits = 1 byte**.
- 총 $4 \times 4 \times 8 = 128$ bits.

### 왜 8-bit 단위인가?

- AES(원래 이름: Rijndael)는 20년 전 **8-bit 마이크로컨트롤러**에서 효율적으로 동작하도록 설계되었다.
- 당시 16-bit 마이크로컨트롤러는 드물었고, 32-bit는 거의 없었다.
- 오늘날 ARM 프로세서는 32-bit 이상이지만, AES의 8-bit 설계는 그대로 유지된다.

## 4. AES 연산들

### 4.1 [[AddRoundKey]]

$$
\text{State}' = \text{State} \oplus \text{RoundKey}
$$

- **Bitwise XOR** 연산.
- 128-bit State와 128-bit Round Key를 XOR.

### 4.2 [[SubBytes]] (S-box)

```
┌─────┐     ┌───────┐     ┌─────┐
│ 8b  │────→│ S-box │────→│ 8b  │
└─────┘     └───────┘     └─────┘
```

- **16개의 동일한 S-box**가 각 바이트에 적용된다.
- 각 S-box는 **8-bit 입력 → 8-bit 출력** 함수.
- **수학적 정의**:
  1. $GF(2^8)$에서 **역원(Inverse)** 계산 (0의 역원은 0으로 정의).
  2. **Affine Transformation** 적용.

### 4.3 [[ShiftRows]]

```
Row 0: [ a  b  c  d ] → [ a  b  c  d ]  (no shift)
Row 1: [ e  f  g  h ] → [ f  g  h  e ]  (left shift by 1)
Row 2: [ i  j  k  l ] → [ k  l  i  j ]  (left shift by 2)
Row 3: [ m  n  o  p ] → [ p  m  n  o ]  (left shift by 3)
```

- **바이트 위치만 변경** (값은 변하지 않음).
- 하드웨어에서는 **단순한 배선(Routing)**이므로 지연이 거의 없다.

### 4.4 [[MixColumns]]

$$
\begin{pmatrix} s'_0 \\ s'_1 \\ s'_2 \\ s'_3 \end{pmatrix}
=
\begin{pmatrix}
02 & 03 & 01 & 01 \\
01 & 02 & 03 & 01 \\
01 & 01 & 02 & 03 \\
03 & 01 & 01 & 02
\end{pmatrix}
\begin{pmatrix} s_0 \\ s_1 \\ s_2 \\ s_3 \end{pmatrix}
$$

- **열(Column) 단위**로 처리.
- **4개의 동일한 MixColumns** 함수가 각 열에 적용.
- **선형(Linear) 연산**.
- 덧셈은 **XOR**, 곱셈은 **$GF(2^8)$에서의 곱셈**.
- 소프트웨어에서 가장 **비용이 많이 드는** 연산 (많은 XOR 필요).

## 5. [[Key Schedule]]

```
     K0   K1   K2   K3
     ↓    ↓    ↓    ↓
    ┌────┬────┬────┬────┐
    │ W0 │ W1 │ W2 │ W3 │  ← Round Key 0 (= 원래 키)
    └──┬─┴──┬─┴──┬─┴──┬─┘
       │    │    │    │
       ↓    ↓    ↓    ↓
    ┌──────────────────┐
    │        g         │ ← 함수 g 적용
    └────────┬─────────┘
             │
    ┌────┬───┴┬────┬────┐
    │ W4 │ W5 │ W6 │ W7 │  ← Round Key 1
    └────┴────┴────┴────┘
             │
            ...
```

### Feistel-like 구조

- $W_4 = W_0 \oplus g(W_3)$
- $W_5 = W_1 \oplus W_4$
- $W_6 = W_2 \oplus W_5$
- $W_7 = W_3 \oplus W_6$

### g 함수

```
┌─────────────────────────────────────┐
│  입력: 32 bits (4 bytes)            │
│       [a, b, c, d]                  │
│                                      │
│  1. Rotate: [b, c, d, a]            │
│  2. SubBytes: S-box를 각 바이트에 적용│
│  3. XOR with Rcon: 첫 바이트에 XOR   │
│                                      │
│  출력: 32 bits                       │
└─────────────────────────────────────┘
```

### [[Rcon]] (Round Constant)

- 각 라운드마다 **다른 상수**.
- 라운드 1: `01`, 라운드 2: `02`, 라운드 3: `04`, ... ($GF(2^8)$에서 2의 거듭제곱)
- **라운드를 구분하는 유일한 요소** (나머지 연산은 모든 라운드에서 동일).

### Key Schedule의 중요한 특성

> **Key Schedule은 가역(Reversible)**이다. **어느 Round Key든 알면 원래 키를 역산할 수 있다.**

- 물리적 공격으로 **마지막 라운드의 Round Key**를 복구하면,
- Key Schedule을 **역으로 계산**하여 **원래 키**를 복구할 수 있다.

## 6. 소프트웨어 구현

### 6.1 기본 구현

- **S-box**: **256-entry Lookup Table**로 구현 (8-bit 입력 → 8-bit 출력).
- **MixColumns**: 가장 비용이 많이 드는 연산 (많은 XOR).

### 6.2 [[T-Table]] 기반 구현

- **아이디어**: 4개의 S-box + 1개의 MixColumns를 **하나의 큰 Lookup Table**로 합친다.
- **T-Table**: 8-bit 입력 → 32-bit 출력.
- **장점**: 더 빠름 (메모리가 충분하다면).
- **단점**: 더 많은 메모리 필요.

```
┌───────────────────────────────────────────┐
│                MixColumns                  │
│  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐      │
│  │S-box│  │S-box│  │S-box│  │S-box│      │
│  └──┬──┘  └──┬──┘  └──┬──┘  └──┬──┘      │
│     │        │        │        │          │
│     ↓        ↓        ↓        ↓          │
│  ┌───────────────────────────────────┐    │
│  │          MixColumns               │    │
│  └───────────────────────────────────┘    │
└───────────────────────────────────────────┘
                    ↓
            ┌─────────────┐
            │  합쳐서     │
            │  T-Table    │
            │  8b → 32b   │
            └─────────────┘
```

### Pseudo Code

```c
// aes_encrypt.c
uint8_t x[16] = plaintext;
x = x XOR round_key[0];

for (round = 1; round <= 10; round++) {
    x = SubBytes(x);
    x = ShiftRows(x);
    if (round != 10) {
        x = MixColumns(x);
    }
    x = x XOR round_key[round];
}

ciphertext = x;
```

## 7. 하드웨어 구현

### 7.1 Round-Based 구현

```
                    ┌────────────────┐
                    │   Multiplexer  │←── Plaintext
                    └───────┬────────┘
                            │ ↑
        ┌───────────────────┴─┴───────────────────┐
        │                                          │
        ↓                                          │
┌───────────────┐                                  │
│   AddRoundKey │←── Round Key                     │
└───────┬───────┘                                  │
        ↓                                          │
┌───────────────┐                                  │
│   SubBytes    │                                  │
└───────┬───────┘                                  │
        ↓                                          │
┌───────────────┐                                  │
│   ShiftRows   │                                  │
└───────┬───────┘                                  │
        ↓                                          │
┌───────────────┐     ┌───────────────┐           │
│   MixColumns  │────→│   Multiplexer │←─ Last Rnd│
└───────────────┘     └───────┬───────┘           │
        ↑                     │                    │
        └─────────────────────┘                    │
                              │                    │
                        ┌─────┴─────┐              │
                        │  Register │──────────────┘
                        └─────┬─────┘
                              ↓
                         Ciphertext
```

- **루프에 레지스터가 필요**하다 (순수 조합 루프는 불가능).
- 레지스터 위치에 따라 **Critical Path Delay**가 결정된다.
- **11 클럭 사이클**에 1회 AES 암호화 완료.

### 레지스터 배치의 중요성

- 레지스터가 **하나**만 있으면 → Critical Path가 길어짐 → 느린 클럭.
- 레지스터를 **여러 개** 배치하면 → Critical Path 단축 → 빠른 클럭 가능 + **파이프라인** 가능.

### 7.2 Unrolling 구현

```
Plaintext → [Round 1] → [Round 2] → ... → [Round 10] → Ciphertext
```

- 모든 라운드를 **조합 회로로 펼침**.
- 레지스터 없음 → **1 클럭 사이클**에 암호화 완료.
- **단점**: Critical Path가 매우 길고, 면적이 매우 큼.

### 7.3 Pipeline Unrolling

```
Plaintext → [Rnd1] → |REG| → [Rnd2] → |REG| → ... → |REG| → [Rnd10] → Ciphertext
```

- 각 라운드 사이에 **레지스터** 배치.
- **Latency**: 10 클럭 사이클.
- **Throughput**: 매 클럭 사이클마다 새로운 평문 입력 가능 → **매우 높은 처리량**.
- **용도**: Cisco 라우터 같은 **고속 네트워크 장비**, **암호 해독기(COPACOBANA)** 등.

### [[COPACOBANA]] (Cost-Optimized Parallel Code Breaker)

- DES **Brute Force 공격**을 위해 설계된 FPGA 기반 장치.
- 120개의 FPGA, 각 FPGA에 8개의 DES 엔진 (Pipeline Unrolling).
- DES 키 크기가 **56 bits**로 짧아서 Brute Force가 가능.
- **높은 Throughput**이 핵심.

### 7.4 Serialized 구현 (면적 최소화)

- **하나의 S-box**만 사용.
- 상태를 **Shift Register**에 저장하고, 한 번에 **1 바이트씩** 처리.
- **면적은 최소**, **시간은 최대**.
- **용도**: 매우 작은 임베디드 시스템.

## 8. FPGA에서의 T-Table 구현

### Block RAM (BRAM)

- FPGA에는 **Block RAM**이라는 내장 메모리가 있다.
- T-Table을 BRAM에 저장하면 효율적인 구현 가능.
- BRAM은 **Dual-Port**를 지원 → 동시에 2개의 주소에서 읽기 가능.
- BRAM 읽기에는 **1 클럭 사이클 지연**이 있다 → 별도의 레지스터가 필요 없다.

# 7️⃣ Summary

## 핵심 개념 정리

| 개념 | 설명 |
|------|------|
| **Kerckhoffs' Principle** | 알고리즘은 공개, 오직 키만 비밀 |
| **Adversary Model** | 보안을 주장하려면 공격자 모델을 정의해야 함 |
| **Cryptographic Device** | 암호 알고리즘 구현 + 키 저장 장치 |
| **Physical Attacks** | 장치의 물리적 특성을 이용한 공격 |
| **Divide and Conquer** | 키 전체가 아닌 부분을 개별적으로 찾는 전략 |

## Physical Attacks 분류

| | Passive | Active |
|---|---------|--------|
| **Invasive** | Probing | Permanent Faults |
| **Semi-Invasive** | Photon Emission | Laser Fault Injection |
| **Non-Invasive** | **Side-Channel Analysis** | **Fault Injection** |

## AES 연산

| 연산 | 설명 | 특징 |
|------|------|------|
| **AddRoundKey** | State ⊕ RoundKey | Bitwise XOR |
| **SubBytes** | 16개의 8→8 S-box | 비선형 연산 |
| **ShiftRows** | 행별 바이트 회전 | 위치만 변경 |
| **MixColumns** | 열별 선형 변환 | 마지막 라운드에 없음 |

## 이 강의의 핵심

> **암호 알고리즘이 아무리 안전해도, 그것을 구현한 장치는 물리적 특성을 통해 비밀 정보를 누출할 수 있다.**

- 실행 시간 → **Timing Attack**
- 전력 소비 → **Power Analysis Attack**
- 외부 조작 → **Fault Injection Attack**
